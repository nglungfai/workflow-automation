Turn a Drive Folder into a Cowork Skill: Auto-Applying the Predactiv Design System

Skill Deep Dive · 6 min read

This site itself just got re-themed with Predactiv's actual brand — Poppins type, the aubergine/teal action colors, pill buttons, 20px card radii — instead of a generic dark theme. The interesting part isn't the re-theme; it's how we made that automatic for every future HTML task, not just this one. If your team ships internal tools, decks, or prototypes and wants them to look like Predactiv without re-explaining the brand every time, this is the pattern to copy.

The problem: a design system sitting in Drive doesn't help unless something reads it

Predactiv already has a real design system — colors_and_type.css with every color, type, radius, and shadow token; logo SVGs; a Figma-sourced brand guide — living in a "Predactiv Design System" folder in Google Drive. Useful, but only if whoever's building something remembers it exists, finds it, and copies the right values in by hand. That doesn't scale past one person.

The fix: package it as an installable Cowork skill

Cowork skills are just a folder with a SKILL.md manifest plus supporting files, zipped with a .skill extension. Once installed via Settings → Capabilities → Skills, Cowork matches the skill's description against your requests and pulls it in automatically — no need to say "use the Predactiv design system" every time. It turns out our Drive folder already had a SKILL.md sitting in it, written for exactly this purpose; it just hadn't been packaged and installed yet.

What we did, step by step

  1. Located the source folder. Searched Google Drive for "Predactiv Design System" and found SKILL.md, README.md, colors_and_type.css, plus an assets/ folder with logo SVGs and bundled Poppins font files.
  2. Pulled exact file contents. This is the part that's easy to get wrong: reading a file's "natural language representation" (what you get from a casual content preview) mangles code syntax — quotes get escaped, symbols get converted to entities. For anything that has to run as real CSS or markdown, you need the exact bytes, which means downloading the raw file content and decoding it, not summarizing it.
  3. Assembled the skill folder. SKILL.md, README.md, and colors_and_type.css copied byte-for-byte, plus the 6 logo SVGs under assets/.
  4. Zipped it with the right structure. The zip needs to contain the skill's folder (matching the name: field in SKILL.md's frontmatter) with SKILL.md inside it — not a flat dump of files, and not the folder's contents zipped without the wrapping folder.
  5. Installed it. Uploaded the .skill file via Settings → Capabilities → Skills. From that point on, any request that looks like building a branded interface, mock, deck, or prototype for Predactiv triggers the skill automatically.

What we deliberately left out — and why

The full Drive folder also has ~30 HTML preview cards (every component rendered standalone) and a full React reconstruction of the Predactiv Data Platform UI under ui_kits/platform/. Those stayed in Drive rather than going into the installed skill, for a simple reason: package size and relevance. The core token file plus logos covers the vast majority of "make this look like Predactiv" requests — plain HTML pages, decks, one-off mocks. The full UI kit is for a narrower case (recreating the actual platform app) and is still just a Drive search away when someone needs it.

We also left out the 8 bundled Poppins .ttf font files — binary assets are impractical to move through a text-based packaging pipeline. colors_and_type.css still references them via local @font-face rules, so if they're genuinely absent, add a Google Fonts fallback link to the page <head> before the stylesheet:

<link rel="stylesheet" href="https://fonts.googleapis.com/css2?family=Poppins:ital,wght@0,400;0,500;0,600;0,700;1,400&display=swap">

A real discrepancy we caught in the process

Worth flagging for whoever owns the design system: README.md and colors_and_type.css disagree on the primary action color. The README describes royal purple #590094 as primary with turquoise #00CBB2 as secondary. The actual CSS file — which labels itself "source of truth" — defines deep aubergine #3F2A7B as primary and muted teal #3D7D7B as secondary. We used the CSS file's values everywhere on this site, since that's what actually renders, but the two documents should be reconciled so the next person pulling from Drive doesn't get conflicting guidance.

Try it yourself

The predactiv-design skill is now installed and available. If you're building an internal tool, a one-off HTML mock, or a deck and want it on-brand without hand-copying hex codes:

  • Just ask Cowork to build it — mentioning "Predactiv" or "branded" is usually enough to trigger the skill.
  • Link colors_and_type.css in your page <head> and use its CSS variables (var(--action-primary), var(--radius-pill), etc.) instead of hardcoding hex values — that's what keeps everything consistent if the tokens ever change.
  • Reach for the existing utility classes (.pdp-btn--primary, .pdp-chip--succeeded, .pdp-card) before rolling your own.
If your team has a similar "the real answer lives in a doc/folder somewhere" problem — a style guide, a glossary, a set of house rules — this same pattern (SKILL.md + supporting files, zipped, installed) is worth copying. See About for how to contribute a follow-up post once you've tried it.
