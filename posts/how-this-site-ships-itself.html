How This Site Ships Itself: Inside an Agent-Run Publishing Pipeline

Case Study · 5 min read

The homepage makes a claim you should be skeptical of: every page here was drafted, committed, and deployed by the same kind of workflow it documents. This post backs it up with one real release — what the AI did, what a human did, what broke, and how it was caught — with a public paper trail you can check: the commit history (the site's save-and-publish log) and the deploy log.

The pipeline: a person describes what should exist; an AI agent writes it; the person approves; the agent publishes it by operating the same website a human contributor would use, clicking the same buttons. A human touches exactly two steps.

One release, 17 published changes

The worked example is this site's biggest release: the overhaul that turned an internal knowledge base into this public portfolio. Every page rewritten; two articles renamed, with every link that pointed at them updated; four files deleted; and three animated illustrations that had a label baked into the pixels edited frame by frame and re-uploaded. Sequencing mattered: new pages went up before anything linked to them, old pages came down last — so mid-release, a visitor never hit a broken site.

The hard part: publishing like a person

The agent has no publishing keys and no direct line to the server — by design, it can only do what a person with a browser could. But GitHub's editor is built for human hands and won't accept text from a program, so the agent does what hands do, synthetically: a "select all" keystroke and a "paste" event, the same signals a keyboard and clipboard produce. For the technically curious:

cm.dispatchEvent(new KeyboardEvent('keydown',
  {key:'a', code:'KeyA', metaKey:true, bubbles:true, cancelable:true}));
const dt = new DataTransfer();
dt.setData('text/plain', newContent);
cm.dispatchEvent(new ClipboardEvent('paste',
  {clipboardData: dt, bubbles: true, cancelable: true}));

The failures here are silent: select the wrong way and the editor quietly pastes a second copy of the page below the first; sometimes the automation reports "failed" when the paste actually worked, so a retry would double the content. The working rule: never trust the report, check the result. After every paste, the agent screenshots the end of the file — right line count, page ends exactly once.

Trust nothing, fingerprint everything

Before saving, the agent computes a fingerprint of what the page should say — a short code where a one-character difference changes everything — and refuses to save unless the rebuilt page matches. It also sweeps for terms that aren't allowed on the public site. That paranoia paid off mid-release: one page came out 216 characters short. The cause was invisible spacing differences left by an old hand edit — harmless, and the fingerprint (built to ignore spacing) proved the content matched, so the release went on with the discrepancy understood rather than ignored. After the final change, the same audit ran against the live site: banned terms zero, renamed pages resolving, retired addresses correctly dead.

The publishing infrastructure failed twice mid-release, too. Neither needed fixing — each publish pushes the entire site, so the last successful one carried everything. The recovery playbook for failures that do need action was learned the hard way once, then written into the agent's instructions for good.

The questions worth asking

"Why click through a website when programmers have better tools?" Safety and generality. The agent's blast radius is a person's blast radius — and most tools worth automating (dashboards, admin consoles, vendor portals) have no programmer's entrance at all. The browser is the universal one; this pipeline is the publicly auditable rehearsal.

"Why not just make 17 changes by hand?" A person could, in an afternoon — but the afternoon repeats every release, and human effort doesn't compound. Every lesson here is written into the agent's standing instructions, so the next release starts smarter. And the agent runs the same checks identically on the first edit and the seventeenth; human attention doesn't.

"Why trust it at all?" You don't — that's the design. Nothing publishes without human approval; nothing is assumed to have worked without checking the result. The trust sits in the verification, not in the AI.

"What about hijacking — the agent reads web pages with a real session?" The risk is prompt injection: a page containing text crafted to look like instructions. The defense is architectural: instructions come only from the operator (page content is data, never commands), the agent works against a fixed set of destinations, and the approval gate is the firebreak — a fooled agent still can't publish anything unsigned. Worst case here is a bad commit to a public repo: visible, revertible. Injection is an open problem industry-wide; anyone claiming immunity is selling something.

"Your checks verify mechanics. Who checks the writing?" A human, deliberately — fingerprints can't prove a paragraph is good. But the gate feeds back: each correction becomes a standing rule the next draft follows, so the reviewer repeats themselves at most once. What's missing is a formal evaluation harness — scored test cases per skill, re-run when models change. That's the known next layer, and the right question to ask any agent system claiming to be production-grade.

The system caught its own mistake

While drafting this very post, an audit of the release's logs found one of the 17 changes had never landed — a button click missed, the edit silently discarded, unflagged for a day because the routine check looked at public pages and this file wasn't one. A verification gap, found by the verification habit, fixed in the same release that shipped this post — and the checklist now reconciles intended changes against the actual log, every release. That's the honest pitch: not that agents don't make mistakes, but that every mistake gets caught by a check, and every check becomes standing procedure. The same error doesn't ship twice.

So what?

A website that publishes itself is a parlor trick. The operating model is not. Swap "publish a page" for bulk-creating audience segments in a live production tool, turning meeting notes into partner-ready documents, or auditing content monthly for drift — the same loop runs them all: describe, draft, gate, execute, verify, keep the lessons. Those systems are private. This one you can audit.
