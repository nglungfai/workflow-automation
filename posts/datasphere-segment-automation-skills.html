Automating DataSphere Segments with Two Cowork Skills

Skill Deep Dive · 7 min read

Bulk segment work in DataSphere — creating dozens of syndicated audiences, or refreshing keywords across a batch of existing ones — used to mean clicking through the same UI flow over and over. We built two Cowork skills to turn that into a review-and-approve task instead: agent-new-syndicated-segments and agent-update-syndicated-segments. Both drive the DataSphere UI directly through Claude in Chrome, so they work exactly like a careful analyst clicking through the app — just faster and consistent.

Prerequisite: Claude in Chrome

Both skills automate the live DataSphere web app rather than calling an API, so they require the Claude in Chrome extension installed and active. If it isn't set up, ask Cowork to trigger either skill and it will tell you what's missing.

Skill 1 — agent-new-syndicated-segments: bulk creation

Use this when you have a list of new segment names to create from scratch, typically sitting in a Google Sheet.

What it needs from you

  • A Google Sheet URL, tab name, and which column holds the segment names
  • A preferred audience size to apply to every segment (or a per-row Size column)
  • Whether the sheet has a header row

What it does, per segment

  1. Reads the sheet (via CSV export, or by reading the live Chrome tab if the export URL fails)
  2. Shows you the parsed segment list and waits for confirmation before touching anything
  3. Asks which taxonomy to create audiences in
  4. For each segment: clicks Create New Audience, names it, clicks Autoscale Topics, sets the preferred size, and saves it as a draft
  5. Reports a clean pass/fail summary at the end — including duplicates it skipped

Skill 2 — agent-update-syndicated-segments: keyword refresh

Use this when segments already exist and need their topics re-autoscaled — for example, after a taxonomy update or a periodic refresh cycle. It does not create anything new.

What it does

  1. Lists audiences in a chosen taxonomy, filtered to Processed status by default, sorted oldest-updated first
  2. Shows you the full table and waits for you to pick which ones to update and what new preferred size(s) to apply
  3. For each selection: clears existing topics, re-runs Autoscale Topics with a fresh description and size, adds all resulting topics, sets the preferred size, and saves as draft
  4. Reports how many topics landed on each audience and flags any that failed

Need both — new segments this quarter, and a refresh on last quarter'sRun them as two separate passes, in either order

Guardrails baked into both skills

These matter because the skills operate on live production audiences:

  • Never "Approve All." Both skills are explicitly instructed to avoid this button — it approves every audience in the taxonomy, not just the ones being touched.
  • Confirm before the bulk loop starts. Both skills show you the full parsed list (segments to create, or audiences to update) and wait for explicit confirmation before making any changes. Mistakes at scale are expensive — review the list.
  • Everything saves as a draft. Nothing goes live automatically; drafts still need your normal review/approval step downstream.
  • Timing matters more than it looks. The update skill waits ~18 seconds for autoscale to generate topics and 5 seconds after adding them before saving — React's state needs time to commit. If a run looks like it's "hanging," that's usually why.

Why this is a good template for your own skill

Both skills follow the same shape worth copying if you're building your own UI-automation skill: collect inputs up front, show the user what you found and get confirmation, execute one item at a time with a verification screenshot after each risky step, and end with an explicit success/failure summary rather than a vague "done." That structure is what makes it safe to run against production data unattended.
