Prompt Engineering vs. Context Engineering vs. Harness Engineering vs. Loop Engineering

Concepts · 7 min read

These four terms get used interchangeably in agent discourse, which is a problem, because they describe four different layers of the same system and each one fails for a different reason. If your Cowork workflow is underperforming, the fix depends on which layer is actually broken. Treating a harness problem as a prompt problem — rewording instructions when the real issue is missing tool access or no verification step — burns turns without fixing anything.

Here's the layer breakdown, roughly in the order the industry adopted the vocabulary: prompt engineering (ChatGPT era, 2022), context engineering (2025, as agents started juggling multi-turn state and long context windows), harness engineering (2026, as the scaffolding around the model became the bottleneck), and loop engineering (mid-2026, for systems that decide on their own when to invoke the agent at all — a term Addy Osmani popularized building on work from Peter Steinberger and Anthropic's Boris Cherny).

Prompt engineering: what to ask

This is the layer everyone starts with and the one this site already covers in depth. It's about a single turn: the wording, structure, and constraints of one request to the model. Specify the deliverable, the audience, the format, what's out of scope. This is necessary but it's the smallest lever in the stack — a perfectly worded prompt still fails if the model doesn't have the right information or the right tools to act on it.

Context engineering: what the model needs to know

Context engineering is about curating what gets loaded into the model's working memory for a given task — which files, which prior turns, which retrieved documents, which schema definitions — so the model can answer confidently without guessing or hallucinating. This matters more than prompt wording once you're past toy tasks, because a context window is a scarce, expensive resource: cramming in everything "just in case" degrades performance as much as leaving out the one fact that mattered.

In our world, this looks like the difference between asking Cowork to "summarize the Tyson audience" cold versus pointing it at the actual Predactiv MCP tools so it pulls live composition data instead of working from stale training knowledge or a half-remembered Slack thread. The prompt didn't change much; what changed is what the model had in front of it when it answered.

Harness engineering: how the whole thing operates reliably

Harness engineering is the operational layer around the model: the toolchain it can call, the constraints on what it's allowed to touch, the feedback loops that tell it whether an action succeeded, and the lifecycle that carries state across multiple context windows rather than a single turn. Where context engineering optimizes what's inside one window, harness engineering is what happens when a task is too big for one window — context resets, structured handoffs between phases, and phase gates that keep a multi-step task coherent as the model's working memory gets wiped and rebuilt.

This is the layer that actually determines whether an agent can be trusted with a real, multi-step, technical task. A skill definition with a scoped tool list, a checklist the agent has to satisfy before moving to the next phase, and a verification step before anything ships — that's harness, not prompt. The agent-pm skill's rule that Slack drafts always route to a review channel before hitting a real recipient is a harness constraint: it's not asking the model nicely to be careful, it's removing the ability to skip the check.

Loop engineering: the system that decides when to prompt at all

Loop engineering is the newest and most autonomous layer: instead of a human typing each prompt, you build the system that decides what work exists, acts on it, verifies the result, remembers what's done, and decides what to do next — on a schedule or continuously, without a human in the loop for each cycle. The agent stops being a tool you hold and becomes a long-running process with its own scheduling and evaluation logic.

A scheduled Cowork task that checks a data pipeline every morning and only pings you when something's actually wrong is a small-scale version of this. The interesting failure mode at this layer isn't a bad response — it's a system that keeps confidently doing the wrong thing on a loop because nothing is verifying its own output before it acts on step two.

Which layer is actually broken?

SymptomLikely layer
Response is vague, wrong format, or misses the deliverablePrompt
Response is confidently wrong or stale despite a clear promptContext
Agent can't complete a multi-step task, loses state, or skips a required checkHarness
A recurring/scheduled agent keeps taking the wrong action unattendedLoop

For a Product org managing model development, audience infrastructure, and partner POCs, the practical implication is where you invest engineering time. Prompt quality is a training problem — coach the team, see Prompting Tips That Actually Move the Needle in Cowork. Context and harness quality are skill-design problems — this is what we're doing every time we scope a Cowork skill's tool list, add a phase gate, or require a review step before something goes external. Loop engineering is where scheduled tasks and always-on monitoring live, and it's the layer that most needs a verification step baked in before you trust it unattended, since nothing is watching it turn by turn.

Try: "Before we automate this as a scheduled task, walk through it as a harness: what tools does it need, what should block it from proceeding to the next step, and what should trigger a human review instead of an autonomous action?"
